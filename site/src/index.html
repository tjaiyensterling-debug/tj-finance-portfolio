The thesis

An AI that's right 95% of the time is unusable in finance if you can't tell which 5% is wrong. So the architecture is always the same three moves — and they generalize far past finance:

01

Compute facts in code

Every number — variances, materiality, margins — is deterministic. The model never produces a figure that lands in a total.

02

Use the model for judgment

Root-cause hypotheses, materiality calls, plain-English narrative for a CFO — the parts an LLM is genuinely good at.

03

Verify before trust

Guardrails and tests that fail loudly — hallucination, coverage, and schema checks; dbt data and unit tests in CI.

Case study · agentic systems

Autonomous, guardrailed multi-agent operations

The projects above prove the discipline on single tasks. This scales it to orchestration: a pipeline of LLM agents — each with a clean context — that runs a recurring, high-stakes workflow, with a person signing off on every irreversible, outward-facing action. The same shape applies to finance ops (e.g. an automated month-end exception-review queue: ingest → score → draft → human approval).

It's a personal tool I run on a daily schedule, so the code stays private — but the architecture is the transferable part, and the engineering decisions behind it are laid out below. Happy to walk through the implementation in detail.

Orchestration

Agents in parallel waves

Independent, fresh-context agents fan out — discover, score, draft, review — and return structured results, not prose, so a deterministic layer can act on them safely.

Guardrails

Untrusted input by default

Every scraped or fetched string is treated as data, never instructions — header-only parsing, URL allow-listing, and a hard human-confirm gate before anything leaves the machine.

Verification

Adversarial self-testing

Independent reviewers attack each change with pre-registered expectations. This caught a prompt-injection gap and a data-loss bug before they reached a live run.

The guardrail, in code (illustrative)

Scraped content is the classic prompt-injection vector. Two cheap rules neutralize it.

import re
from urllib.parse import urlparse

ALLOWED_HOSTS = {"example-source.com", "example-system.com"}  # explicit allow-list

# A scraped record is DATA, not instructions.
# Parse only the trusted header; ignore the free-text body entirely.
def parse_record(md: str) -> dict:
    fields = {}
    for line in md.splitlines():
        if line.strip() in ("---", "") or line.startswith("## "):
            break                       # stop at the untrusted body
        key, _, val = line.partition(":")
        fields[key.strip().lower()] = val.strip()
    return fields

# A URL only becomes a clickable action if its host is on the allow-list;
# otherwise it's surfaced for human verification, never auto-followed.
def vet_url(u: str) -> bool:
    return bool(re.match(r"https?://", u)) and urlparse(u).hostname in ALLOWED_HOSTS

What it demonstrates

For an AI-native finance or operations role: I can take LLM agents past demos into a real, self-operating system — one with explicit trust boundaries, a deterministic core the model's output can't silently override, a human gate on anything irreversible, and an adversarial test loop that finds its own failures before they reach a live run. Same instinct as the variance agent above (which 5% is wrong, and how do I catch it) — applied to autonomy.
