The thesis

An AI that's right 95% of the time is unusable in finance if you can't tell which 5% is wrong. So the architecture is always the same three moves — and they generalize far past finance:

01

Compute facts in code

Every number — variances, materiality, margins — is deterministic. The model never produces a figure that lands in a total.

02

Use the model for judgment

Root-cause hypotheses, materiality calls, plain-English narrative for a CFO — the parts an LLM is genuinely good at.

03

Verify before trust

Guardrails and tests that fail loudly — hallucination, coverage, and schema checks; dbt data and unit tests in CI.

The projects

Both are synthetic-data only, clone-and-build, and verifiable — no warehouse account required.

Claude Finance Variance Agent

Python · Anthropic SDK

Two periods of GL line items in → a CFO-ready variance narrative + structured exception flags out. Variance math is deterministic; the model is used only for judgment and language.

  • • Hallucination check — named accounts must exist in source
  • • Coverage check — every material variance addressed
  • • Schema check — JSON validates or the run aborts
View repo →

dbt Finance Variance

dbt · DuckDB

The same logic, modern-data-stack way: sources → staging → marts, governed by two tiers of tests. dbt build runs locally in under a minute.

  • • Data tests — not_null, unique, accepted_values, relationships
  • • Unit test — pins variance math & the materiality boundary in CI
  • • Configurable — threshold & periods via --vars
View repo →

dbt GPU Cost Attribution

dbt · DuckDB · FinOps

The same discipline, taken into AI unit economics: attribute shared GPU cost to tenants and compute per-tenant margin. Idle capacity is absorbed like factory overhead.

  • • Absorption key — idle GPU cost split by token share
  • • Three test tiers — data + unit + singular tests
  • • Surfaces the loss — flags the unprofitable tenant by margin zone
